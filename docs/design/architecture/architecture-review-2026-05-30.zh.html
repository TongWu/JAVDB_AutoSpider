1 · 折叠 Rust↔Python 的 parity 影子实现

接缝(seam)后面顶着两份可能并无差异的完整实现。押定 Rust,删掉 Python 双胞胎。

涉及文件

  • javdb/parsing/fallback/{index,detail,tag}_parser.py — 1,070
  • javdb/proxy/pool.py + ban_manager.py — 942
  • javdb/spider/url_helper.py · parsing/magnet_categorize.py · infra/masking.py — 814
  • 由 tests/parity/ + tests/unit/test_magnet_parity.py 守护

现状 —— 浅接缝、双份实现

改造后 —— 接缝后只剩一份实现

⇣

预编译 wheel = 硬依赖
Python 影子 + parity 测试退役
删除 ~2,826 行

问题

两个适配器只有在"行为跨接缝有差异"时才构成真正的接缝 —— 而这里两份被要求完全一致,靠 parity 测试强制保证。这个接缝顶着的是重复的局部性(locality),不是变化。

方案

把 Rust wheel 设为硬依赖(CI 已在构建它)。删除 Python 重实现;把 import fallback 收成一个清晰的"需要 Rust core"报错。

收益

  • 局部性(locality):每个行为只有一处
  • 删除 ~2,826 行 + parity 测试套件
  • 反向删除测试(deletion test):删掉 Python,复杂度不会重现
  • 整类漂移消失(不再有跨语言同步)
需要权衡的取舍:Python 影子换来纯 Python 可移植性(无需 Rust 工具链即可运行)。如果该 fallback 对某个部署目标确实是承重的,这条就从"强烈推荐"降为"值得探索" —— 这正是 grilling 的第一个问题。
值得探索 进程内 · PyO3 幽灵适配器

问题

requester 接缝有一个在用适配器(Python)和一个死适配器(Rust,只在一个状态标志里露面)。幽灵适配器提供零杠杆(leverage),却看起来像一个其实不存在的选项。

方案

要么把 RustRequestHandler 接成真实抓取路径、退役 Python 级联;要么删掉死适配器、让 Python 成为规范实现。不要两者都留。

收益

  • 干掉一个幽灵接缝
  • 一份抓取实现,一处可测
  • 方案 B 成本低、今天就能删死代码
  • 方案 A 把 TLS / CF-bypass 逻辑集中到 Rust
为何只给"值得探索":这条路径是 I/O 密集(见卡片 4)—— Rust 在这里不提速。收益是局部性,不是吞吐。而且 curl_cffi 的 JA3/TLS 伪装是绕 Cloudflare 的承重技巧;reqwest 不易复刻它。方案 B(删幽灵)是安全的即时动作;方案 A 是一个真正的研究型项目。
推测性 常驻服务 · Go ⚠ 与 ADR-017 冲突

问题

真正的摩擦不在语言 —— 而在于 D1 查询 + 鉴权 + 响应格式已经在 Python 和 TS 之间手工重复(一个被人手复制的浅接缝)。Go 会给同一份逻辑再加第三处。

方案

真要动:先把 重合面 深化为一份共享契约(扩展 ADR-018 Contract Golden),在换语言之前—— 或者干脆代替换语言。

⚠ 与 ADR-017(双后端同步)冲突 —— 但值得点名,因为 ADR-017 靠纪律管理的那份重复,本身就是这里的深度问题。Go 重写会让纪律更难维持,而非更容易。只有当你想直接攻击这份重复时,才重开 ADR-017/ADR-018;否则保持 Python 不动。
不要重写 跨进程并发

4 · 爬虫热循环 —— 看似最该重写、其实不是

"用 Go 重写爬虫提并发"是最诱人的动作。但数字说不。

单次页面抓取,按耗时拆解

throttle + sleep  (6–20s,GIL 已释放)
网络 I/O  (GIL 已释放)
解析
← 几乎所有 wall-clock 都是 sleep + 网络(I/O 密集) Rust 解析 ≈ 一条细缝 →
并发是跨进程实现的 —— GitHub Actions runner 矩阵,由 Cloudflare RunnerRegistry / MovieClaim Durable Object 协调 —— 而不是在单进程内。HTML 解析早已下放到 Rust。GIL 从不限制吞吐。

为何不

  • I/O 密集:线程早已释放 GIL
  • CPU 细缝(解析)已经是 Rust
  • 扩展 = 加 runner,不是加线程

应该做

  • 加 runner 分片(跨进程)
  • 确保装上 Rust wheel(见卡片 1)
  • 调 coordinator throttle,而非换语言

首要建议

从 卡片 1 —— 折叠 Rust↔Python parity 影子 开始。

它杠杆最高、风险最低,而且直接回答了问题:值得放进 Rust 的部分已经在 Rust 里了。要做的是把迁移收尾 —— 删掉 ~2,826 行 Python 影子,以及把它们强行锁在一起的 parity 测试套件 —— 把一个双份的浅接缝,变成一个单一的深接缝。卡片 2(删幽灵 requester)是自然的下一步。卡片 3 在 ADR-017 后面停泊,卡片 4 是一个刻意的"不要"。

决定全局的唯一问题:纯 Python fallback 现在是否还是一个真实的部署目标?grilling 应该从这里开始。
