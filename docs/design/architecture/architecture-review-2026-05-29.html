Candidate A

Give the integrations a real seam — stop aliasing modules

Strong ports & adapters

apps/cli/{qb,pikpak,rclone,notify}/*.py · javdb/integrations/{qb,pikpak,rclone,notify}/*.py · apps/api/routers/operations.py

Before — the seam is collapsed

  API["apps.api.routers.operations"]
  IMP["javdb.integrations.notify.email
argparse · main() · sys.exit · SMTP · git"]

The CLI module becomes the integration module. No interface sits between them. The integration owns argparse + sys.exit, so the API can't reuse it in-process.

After — one Options→Result interface

flowchart TB
  CLI2["apps.cli.notify.email
parse args · raise SystemExit(code)"]
  API2["apps.api.routers.operations"]
  SVC["javdb.integrations.notify
service.run(Options) → Result"]
  CLI2 -- Options --> SVC
  API2 -- Options --> SVC
  classDef deep fill:#0f172a,color:#e2e8f0,stroke:#0f172a;
  class SVC deep

Two adapters (CLI + API) now justify the seam. Domain logic sits behind a typed Options→Result; process concerns (argparse, exit codes) live only in the CLI adapter.

Problem

Five CLI files do sys.modules[__name__] = _module — the entrypoint is a pass-through, and the integration owns main()/sys.exit(), so callers can't reuse it.

Solution

Move argparse/exit into the CLI adapter; expose a typed service.run(Options)→Result from each integration. Exactly what ADR-015 already specifies.

  • leverage: one interface, two adapters
  • locality: process concerns leave the library
  • API reuses integrations in-process
  • tests hit Options→Result, not sys.exit
✓ Aligns with ADR-015 (Accepted, 0/7 IMPs built). The diagnosis, phased plan, and architecture guards are already written — this is execution, not new design.

~15–20 D1 queries are parallel-maintained. JWT signing matches too — except Python tracks revoked JTIs and the TS backend has no revocation path: a silent security drift.

After — one shared definition, two thin readers

openapi.json is already generated from the Python app and consumed one-way by TS for types. Push that further: a real seam both backends read from, instead of two hand-copied implementations.

Problem

Auth rules and ~15–20 D1 queries exist verbatim in both backends; the only thing keeping them aligned is a prose rule. Revocation logic has already drifted.

Solution

Lift the overlapping queries + auth contract into a shared seam (shared .sql + codegen, or generate one backend from openapi.json). Both backends become adapters over it.

  • locality: query change in one place
  • closes the JWT revocation drift
  • deletes the manual "sync both" rule
  • leverage: contract drives both adapters
⚠ Reopens ADR-017's two-backend split. The split itself is sound (Python executes, Worker dispatches) — but the shared query surface was left as manual sync. Worth a new ADR to pick the seam (shared SQL vs codegen vs Worker-proxies-Python). Heavier than A; the security drift makes it more than cosmetic.

Candidate C

Make the Repo deep — retire the db_* function facade

Strong in-process

javdb/storage/repos/history_repo.py · javdb/storage/db/_db_history_write.py · javdb/storage/db/_db_session.py

One interface, one test surface. session_id bound at construction and mandatory — the global sentinel disappears.

Problem

The Repo classes (ADR-005's intended interface) delegate straight to module-level db_* functions that are also public. Two ways to write history; tests patch the functions; session_id leaks to a global.

Solution

Move the implementation behind the Repo, make db_* private, bind session_id at construction and require it. Finish what ADR-005 started.

  • interface shrinks; implementation absorbs it
  • locality: one place to evolve writes
  • kills implicit global session state
  • tests cross one seam, not two
✓ Completes ADR-005's direction. The Repo seam was introduced to kill the old db.py god-module; the function facade is the last remnant of the old interface still exposed alongside it.

Candidate E

Own the session lifecycle in one module

Worth exploring in-process

javdb/storage/db/_db_reports.py · _db_history_write.py · javdb/storage/sessions/commit.py · javdb/storage/rollback/core.py

Before — the state machine is spread thin

_db_reports · read Status/WriteMode
_db_history_write · flip → finalizing/committed
sessions/commit.py · parallel commit path
rollback/core.py · decide resume vs roll back

No single owner of "which transition is legal." Callers can flip status out of order; commit orchestration is 1000+ lines across 6 functions.

After — one band owns the transitions

SessionLifecycle

can_transition · transition · get_state

in_progress → finalizing → committed / failed

commit + rollback read this one interface

Illegal transitions become impossible to express; commit/rollback share one validator.

Problem

The in_progress→finalizing→committed/failed lifecycle is mutated from 4+ modules with no shared validator; commit is a 1000-line procedure only testable against a real DB.

Solution

A SessionLifecycle module owning state queries + transitions, plus a CommitPipeline of small testable steps behind it.

  • locality: transitions in one module
  • illegal transitions unrepresentable
  • commit steps testable in isolation
  • rollback + commit share one seam

Problem

Lowest urgency — leaks are mostly in logging/recovery, and the SQLite mirror is being retired (per CONTEXT.md / ADR-010). Worth folding in opportunistically rather than as its own project.

Top recommendation

Start with Candidate A — the integrations seam (ADR-015).

It is the only candidate where the design is already accepted and fully planned (7 phased IMPs + architecture guards written), so there is no design risk — just execution. The friction is concrete and verified: five CLI files alias their integration module via sys.modules[__name__] = _module, and the integrations own argparse/sys.exit, which is why the API and the pipeline can't reuse them in-process. Deepening here creates a real Options→Result seam with two adapters (CLI + API), which in turn unblocks killing the subprocess-only pipeline coupling.

Then Candidate C (finish ADR-005's Repo by retiring the db_* facade) for the highest testability gain in the storage core. Treat Candidate B (dual-backend) as its own ADR — it carries a real security drift, but the seam choice deserves a decision before code.
