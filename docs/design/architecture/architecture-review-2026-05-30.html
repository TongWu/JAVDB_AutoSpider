1 · Collapse the Rust↔Python parity shadow

The seam fronts two full implementations that may not differ. Commit to Rust; delete the Python twin.

After — one implementation behind the seam

⇣

Python shadow + parity tests retired

Problem

Two adapters justify a seam only when behaviour varies across it — here both are required to be identical, enforced by parity tests. The seam fronts duplicated locality, not variation.

Solution

Ship the Rust wheel as a hard dependency (CI already builds it). Delete the Python re-implementations; reduce the import-fallback to a clear "Rust core required" error.

Wins

  • locality: one place per behaviour
  • delete ~2,826 LOC + parity suite
  • inverted deletion test: drop Python, complexity does not reappear
  • drift class disappears (no two-language lockstep)
Tradeoff to weigh: the Python shadow buys pure-Python portability (run with no Rust toolchain). If that fallback is genuinely load-bearing for some deploy target, this becomes "Worth exploring" rather than Strong — the first grilling question.

Problem

The requester seam has one live adapter (Python) and one dead one (Rust, surfaced only in a status flag). A phantom adapter gives zero leverage and reads as a real option that isn't.

Solution

Either wire RustRequestHandler as the real fetch path and retire the Python cascade, or delete the dead adapter and make Python canonical. Don't keep both.

Wins

  • kill a phantom seam
  • one fetch implementation, one place to test
  • Option B is cheap and removes dead code today
  • Option A concentrates TLS/CF-bypass logic in Rust
Why only "Worth exploring": this path is I/O-bound (Card 4) — Rust buys no throughput here. The win is locality, not speed. And curl_cffi's JA3/TLS impersonation is the load-bearing trick against Cloudflare; reqwest does not replicate it trivially. Option B (delete the phantom) is the safe, immediate move; Option A is a real research project.

Problem

The real friction isn't the language — it's that D1 queries + auth + response shapes are duplicated across Python and TS already (a shallow seam replicated by hand). Go adds a third site of the same logic.

Solution

If touched at all: deepen the overlapping surface into one shared contract (extend ADR-018 Contract Golden) before — or instead of — changing language.

⚠ Contradicts ADR-017 (dual-backend sync) — but worth naming because the duplication ADR-017 manages by discipline is itself the depth problem. A Go rewrite makes the discipline harder, not easier. Reopen ADR-017/ADR-018 only if you want to attack the duplication directly; otherwise leave Python in place.
Do not rewrite cross-process concurrency

4 · The spider hot loop — the obvious candidate that isn't

"Rewrite the scraper in Go for concurrency" is the tempting move. The numbers say no.

One page-scrape, by time spent

throttle + sleep  (6–20s, GIL released)
network I/O  (GIL released)
parse
← almost all wall-clock is sleep + network (I/O-bound) Rust parse ≈ a sliver →
Parallelism is achieved across processes — GitHub Actions runner matrix, coordinated by the Cloudflare RunnerRegistry / MovieClaim Durable Objects — not within one process. HTML parsing is already offloaded to Rust. The GIL never gates throughput.

Why not

  • I/O-bound: threads already release the GIL
  • CPU sliver (parse) is already Rust
  • scale = more runners, not more threads

Instead

  • add runner shards (cross-process)
  • ensure the Rust wheel is installed (Card 1)
  • tune coordinator throttle, not the language

Top recommendation

Start with Card 1 — collapse the Rust↔Python parity shadow.

It is the highest leverage, lowest risk move and it directly answers the question: the parts worth having in Rust are already in Rust. The work is to finish the migration — delete ~2,826 lines of Python shadow and the parity suite that holds them in lockstep — turning a doubled, shallow seam into a single deep one. Card 2 (delete the phantom requester) is the natural follow-on. Card 3 stays parked behind ADR-017, and Card 4 is a deliberate "don't."

The one decision that gates everything: is the pure-Python fallback still a real deploy target? That's where the grilling should start.
