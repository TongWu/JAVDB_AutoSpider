两个适配器（CLI + API）现在让接缝成立。领域逻辑藏在带类型的 Options→Result 之后；进程层关切（argparse、退出码）只活在 CLI 适配器里。

问题

五个 CLI 文件执行 sys.modules[__name__] = _module —— 入口只是个穿透；而 integration 自带 main()/sys.exit()，调用方无法复用。

方案

把 argparse/退出码移进 CLI 适配器；让每个 integration 暴露带类型的 service.run(Options)→Result。这正是 ADR-015 已经规定好的。

  • 杠杆（leverage）：一个接口，两个适配器
  • 局部性（locality）：进程关切移出库
  • API 可在进程内复用 integrations
  • 测试打到 Options→Result，而非 sys.exit
✓ 与 ADR-015 一致（已接受，0/7 个 IMP 已建）。诊断、分阶段计划、架构守卫都已写好 —— 这是执行，不是新设计。

候选 E

让 session 生命周期归一个模块所有

Worth exploring（值得探索） in-process

javdb/storage/db/_db_reports.py · _db_history_write.py · javdb/storage/sessions/commit.py · javdb/storage/rollback/core.py

之前 — 状态机被摊薄铺开

_db_reports · 读 Status/WriteMode
_db_history_write · 翻转 → finalizing/committed
sessions/commit.py · 并行提交路径
rollback/core.py · 决定 resume 还是 roll back

没有单一所有者来管"哪个转换是合法的"。调用方可能乱序翻转状态；commit 编排是跨 6 个函数的 1000+ 行。

之后 — 一个条带拥有所有转换

SessionLifecycle

can_transition · transition · get_state

in_progress → finalizing → committed / failed

commit + rollback 都读这一个接口

非法转换变得无法表达；commit/rollback 共享一个校验器。

问题

in_progress→finalizing→committed/failed 生命周期被 4+ 个模块修改，没有共享校验器；commit 是一个 1000 行的过程，只能对着真实 DB 测试。

方案

用一个 SessionLifecycle 模块拥有状态查询 + 转换，再加一个由小而可测步骤组成的 CommitPipeline 藏在它之后。

  • 局部性：转换集中在一个模块
  • 非法转换无法表达
  • commit 步骤可隔离测试
  • rollback + commit 共享一个接缝

首选推荐

从候选 A 开始 —— integrations 接缝（ADR-015）。

它是唯一一个设计已接受且已完整规划的候选（7 个分阶段 IMP + 架构守卫均已写好），因此没有设计风险 —— 只剩执行。摩擦具体且已核实：五个 CLI 文件通过 sys.modules[__name__] = _module 别名其 integration 模块，而 integrations 自带 argparse/sys.exit，这正是 API 和 pipeline 无法进程内复用它们的原因。在这里深化会造出一个真正的 Options→Result 接缝、带两个适配器（CLI + API），进而解开 pipeline 仅靠 subprocess 的耦合。

之后做候选 C（退役 db_* 门面、完成 ADR-005 的 Repo），它在存储核心带来最高的可测试性收益。把候选 B（双后端）当作独立 ADR 处理 —— 它带着真实的安全漂移，但接缝怎么选值得先做决定再动代码。
